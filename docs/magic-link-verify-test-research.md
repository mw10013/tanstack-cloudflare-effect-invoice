# Magic Link Verify Test Research

## Question

In `test/integration/smoke.test.ts`, should the test:

- rewrite the generated `magicLink` URL to `example.com`
- use `redirect: "manual"`
- or just request the generated magic link as-is and let it redirect naturally

## Short Answer

- Rewriting the generated `magicLink` host/protocol/port is suspicious. It changes the URL under test.
- `redirect: "manual"` is only justified if the test explicitly wants the intermediate HTTP redirect response and its `Set-Cookie` header.
- If the intent is "do what production does", the test should use the generated magic link as-is and follow redirects like a browser.
- If the intent is "inspect the auth redirect hop and capture the cookie manually", then `redirect: "manual"` is a harness technique, not product behavior.

## Ground Facts

### 1. The integration harness uses `example.com` as a placeholder base URL

`test/test-utils.ts` builds server-fn RPC requests with `http://example.com`:

```ts
const fetchServerFn = (url: string, init?: RequestInit) =>
  exports.default.fetch(
    new Request(new URL(url, "http://example.com"), init),
  );
```

Source: `test/test-utils.ts:42-45`

Existing repo notes already describe integration tests this way:

```md
- Integration tests | `test/integration/` | Use fixed `http://example.com`, no localhost ports.
```

Source: `docs/archive/multiple-clones.md:52-58`

So `example.com` is test harness scaffolding for relative worker requests. It is not an application requirement.

### 2. The generated magic link comes from Better Auth `baseURL`

The app config passes `BETTER_AUTH_URL` into Better Auth as `baseURL`:

```ts
const options = {

```

Source: `src/lib/Auth.ts:98-100`

Local Wrangler config sets:


Source: `src/lib/Login.ts:26-31`, `src/lib/Login.ts:37-41`

So the returned `magicLink` is not fake. It is the exact Better Auth link generated from configured app auth origin.

Bundled implementation matches that doc exactly:


Source: `node_modules/.pnpm/better-auth@1.5.1_fd1b5e0f29a90e22d31e3f2a15cb9b37/node_modules/better-auth/dist/plugins/magic-link/index.mjs:156-161`

So if the test uses the generated link unchanged, a redirect is expected. That is not suspicious by itself.

### 4. Better Call implements redirect as an `APIError`

```js
redirect: (url) => {
  headers.set("location", url)
  return new APIError("FOUND", void 0, headers)
}
```

Source: `node_modules/.pnpm/better-call@1.3.2_zod@4.3.6/node_modules/better-call/dist/context.mjs:61-63`

Router handling later converts `APIError` to a `Response`:

```js
if (isAPIError(error)) return toResponse(error)
```

Source: `node_modules/.pnpm/better-call@1.3.2_zod@4.3.6/node_modules/better-call/dist/router.mjs:89-93`

That explains the observed unhandled rejection shape during the test:

- status `FOUND`
- statusCode `302`
- name `APIError`

The error is the redirect mechanism.

### 5. `redirect: "manual"` changes what the test is asserting

MDN on `Request.redirect`:

```md
A `RequestRedirect` enum value, which can be one the following strings:

- `follow`
- `error`
- `manual`

If not specified when the request is created, it takes the default value of `follow`.
```

Source: MDN `Request: redirect property`

So `redirect: "manual"` means: do not behave like the default browser-style fetch flow. Give the caller the intermediate redirect response instead.

That is useful here only if the test wants to inspect:

- the `302` status
- the `Location` header
- the `Set-Cookie` header on the redirect response

The current helper proves that is exactly what the test is doing:

```ts
export function extractSessionCookie(response: Response): string {
  const setCookieHeader = response.headers.get("Set-Cookie");
  ...
  return `better-auth.session_token=${match[1]}`;
}
```

Source: `test/test-utils.ts:71-77`

So `redirect: "manual"` is not required by Better Auth. It is required by this test shape because the test is manually extracting the cookie from the intermediate response.

## What Looks Wrong vs What Looks Reasonable

### Reasonable

- Using `redirect: "manual"` when the test's goal is to assert raw HTTP redirect behavior.
- Using `redirect: "manual"` when the test needs the redirect response's `Set-Cookie` header to manually simulate a cookie jar.

### Suspicious

- Rewriting the generated magic-link origin from `localhost:3000` to `example.com`.
- Deleting `callbackURL` from the generated link just to avoid the redirect path.

Those both alter the URL under test.

## Why `example.com` Rewriting Feels Wrong

The app under test generated a concrete link. The most black-box integration assertion is to request that link exactly.

The current worker code does not make the hostname look special for this path. It checks `pathname`:

```ts
const url = new URL(request.url);
const isMagicLinkRequest =
  (url.pathname === "/login" && request.method === "POST") ||
  url.pathname === "/api/auth/magic-link/verify";
```

Source: `src/worker.ts:255-263`

That does not prove host rewriting is harmless everywhere. It only shows there is no obvious worker-side requirement that the verify request be changed to `example.com`.

Given that the generated link is already absolute, the safer research conclusion is:

- prefer requesting `result.magicLink` unchanged
- only normalize the origin if a concrete test-harness limitation is demonstrated

No such limitation has been demonstrated yet.

## Why `redirect: "manual"` Feels Suspicious But Can Still Be Legit

It depends on the test goal.

If the goal is:

> "simulate a user clicking the email link and ending up authenticated"

then `redirect: "manual"` is the wrong abstraction level. A browser follows redirects and manages cookies automatically.

If the goal is:

> "assert the exact HTTP redirect and manually capture the session cookie"

then `redirect: "manual"` is appropriate. But that should be called out explicitly as an HTTP-level test, not a browser-behavior test.

## Practical Conclusion

For this repo, the cleanest mental model is:

- `example.com` is a test harness placeholder for relative worker requests.
- `magicLink` is an app-generated absolute URL and should generally be treated as the artifact under test.
- `redirect: "manual"` is a fetch-harness choice for inspecting intermediate redirect responses.
- The unhandled rejection is consistent with Better Auth's redirect implementation, not proof that the generated URL is malformed.

## Recommendation

Do not change the generated magic link in order to make the test pass unless a concrete harness limitation is proven.

Choose one of two test styles and keep them explicit:

1. Browser-style test

- Request the generated `magicLink` as-is.
- Let redirects happen.
- Assert on the final authenticated state.
- Best fit for Playwright or a helper that follows redirects with cookie persistence.

2. Raw HTTP auth-hop test

- Request the generated `magicLink` as-is.
- Use `redirect: "manual"`.
- Assert `302`, `Location`, and `Set-Cookie`.
- Manually pass the session cookie into the next request.

What should be avoided is the hybrid approach:

- rewriting the generated URL
- stripping `callbackURL`
- then claiming the test still models the production click path

That is the part that is legitimately sneaky.
